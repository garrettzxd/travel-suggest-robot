# 行程规划卡片（Itinerary Card）— 需求与技术方案

> 设计稿基准：[plans/2026-04-26/Itinerary-planning.html](Itinerary-planning.html)

---

## 一、需求描述

### 1.1 当前现状

目前「漫游助手」已有四类结构化产出，组合渲染为一张 TripCard：
- 地点 Hero（城市 + 评估 pill + 顶图）
- 天气子卡（实时 + 7 日预报 + 整体评估）
- 景点列表（6–8 条，含图、评分、距离）
- 出行建议面板 + 4 条追问 chips

但当用户问"如何规划行程 / 三天两晚怎么玩 / 五一假期怎么安排"等**纯行程编排类问题**时，bot 会走自然语言文本回答，前端只能以 markdown 方式展示一段长文。问题：
1. 文本不结构化，"上午 / 下午 / 晚上"层次靠 markdown 列表手撸，节奏与重点不直观；
2. 每天主题、关键节点、贴士、备选活动等信息混在一起，扫读体验差；
3. 已有 TripCard 的视觉语言（卡片化、左侧 rail + 右侧时间轴）没有复用，UI 分裂。

### 1.2 迭代功能目标

新增一张**行程规划卡片**（Itinerary Card），与 TripCard 平级共存：
- 当用户问行程编排类问题时，bot 输出**结构化的逐日行程**，前端以独立卡片形式渲染，按 Day 1 / Day 2 / … 分组，每天若干时段（上午/下午/傍晚 或 具体时间）的活动条目。
- 卡片样式与文案严格按设计稿 [Itinerary-planning.html](Itinerary-planning.html) 还原（宋体标题、左侧 DAY rail、时间轴节点、accent 色 = `#19C4FD`、虚线分隔的提示区）。
- 与 TripCard 互斥但不冲突：单轮回复要么是 TripCard（用户想去某地），要么是 ItineraryCard（用户想要行程编排），要么纯文本（其他咨询）。后续可在同一会话不同回合中先后出现。

### 1.3 整体交互流程

1. 用户在聊天输入框发问，例如：
   - "杭州三天两晚怎么规划？"
   - "五一假期想去成都玩 4 天，能帮我安排吗？"
   - "想带老人去苏州两天，行程怎么安排比较轻松？"
2. 后端识别为"行程编排"意图后，调用新工具产出结构化行程数据（不是 markdown 文本）。
3. 通过 SSE 推送一个新事件 `itinerary` 给前端，事件 payload 是结构化的逐日行程对象。
4. 前端 SSE 客户端收到 `itinerary` 事件后，把数据 patch 到当前 assistant 消息上，触发渲染条件切换，展示 ItineraryCard。
5. 卡片渲染完毕后，本轮回复以空字符串收尾，与 TripCard 一致（不再追加 markdown 文本）。
6. 卡片下方仍可继续追问（沿用现有"基于已有卡片的追问，本轮不调工具，自然语言回答"规则）。

边界情况：
- 用户没有给出明确目的地（例如"五一去哪好"）→ 不出 ItineraryCard，仍走自然语言候选目的地建议。
- 用户给了目的地但没说天数（"杭州怎么玩"）→ 由模型自行选择合理天数（默认 2–3 天）并在卡片标题里写明。
- 用户给了目的地 + 想看天气和景点（"我想去杭州"）→ 仍走 TripCard，不走 ItineraryCard。两条意图分支由 prompt 触发条件区分。
- 极端天气、人群适配（带老人/带娃）等约束，反映在每日 `subtitle` 与 `footnote` 里，不另开字段。

## 二、Backend 技术方案

### 2.1 共享类型（packages/shared）

在 [packages/shared/src/travel.ts](../../packages/shared/src/travel.ts) 末尾新增 `Itinerary` 系列类型定义，与 `TripCard` 平级；`index.ts` 已经走 `export *` 自动透出，无需改入口。

字段严格对齐设计稿渲染需要：

```ts
/**
 * 一日行程内的单个活动条目。
 * 对应设计稿里时间轴上的一个节点。
 */
export interface ItineraryItem {
  /** 时段或具体时间，如 "上午"、"傍晚"、"8:00"、"或"（备选项前缀）；可空 */
  time?: string;
  /** 主活动标题，如 "断桥残雪 → 白堤 → 平湖秋月" */
  title: string;
  /** 右侧标签 pill，如 "步行"、"船游"、"人文"、"城市漫步"；可空 */
  tag?: string;
  /** 一句话补充说明：路线、时长、提示 */
  desc: string;
}

/**
 * 一日行程：左侧 rail 显示 "DAY N + subtitle"，右侧时间轴串多条 item。
 */
export interface ItineraryDay {
  /** 当天主题副标题，如 "西湖核心区"、"灵隐 + 西溪 / 龙井"；可空 */
  subtitle?: string;
  /** 时间轴条目，建议 2–5 条 */
  items: ItineraryItem[];
}

/**
 * 一张完整的「行程规划卡」。由新工具 recommendItinerary 产出，
 * 前端通过 SSE `itinerary` 事件接收并渲染。
 */
export interface Itinerary {
  /** 卡片标题，如 "5/1 三天两晚 · 杭州行程" */
  title: string;
  /** 逐日时间轴，长度 = 总天数（建议 1–5 天） */
  days: ItineraryDay[];
  /** 底部"提示"虚线区文案；可空 */
  footnote?: string;
}
```

> **字段命名与设计稿一一对齐**（`title` / `days` / `subtitle` / `items` / `time` / `title` / `tag` / `desc` / `footnote`），避免前端再做字段映射。  
> **不引入** 设计稿里没有视觉区分的 `highlight` 字段，避免无效字段污染 schema。

### 2.2 新工具：recommendItinerary

新建 [apps/server/src/agent/tools/recommendItinerary.ts](../../apps/server/src/agent/tools/recommendItinerary.ts)，参照 [finalizeTripCard.ts](../../apps/server/src/agent/tools/finalizeTripCard.ts) 的结构：

- 用 LangChain `tool(...)` + Zod schema 定义；schema 字段一对一镜像 `Itinerary` interface（`title` / `days[].subtitle?` / `days[].items[]` / `footnote?`），并按 finalizeTripCard 的"边界注释 + JSDoc"风格写好工具说明，让模型知道调用时机与字段约束。
- 校验约束（在 schema 里直接表达）：`days.length` 1–5；每个 `day.items.length` 1–8；`title` 非空、长度限制 < 60；`tag` 长度 < 8 防止溢出 pill。
- 工具 `func` 直接 `return JSON.stringify(input)` —— 与 getAttractions / finalizeTripCard 一致，规避 Moonshot 对非字符串返回值的 "unknown content type" 错误。
- 工具命名导出 `recommendItineraryTool`，类型 `ToolName` 增加 `'recommendItinerary'` 字面量（如已枚举集中维护）。

### 2.3 注册到 LangGraph

修改 [apps/server/src/agent/graph.ts](../../apps/server/src/agent/graph.ts) L12–16，把 `recommendItineraryTool` 加入 `tools` 数组，与 `finalizeTripCardTool` 同样属于"内部工具"——不下发 `tool_start` / `tool_end`，由 chat 路由处理为新事件 `itinerary`。

### 2.4 SSE 路由扩展

修改 [apps/server/src/routes/chat.ts](../../apps/server/src/routes/chat.ts)：

1. **新增事件类型**：在 SSE 事件枚举里追加 `'itinerary'`（参考 `'card'` 现有定义）。payload 形状：`{ itinerary: Itinerary }`。
2. **on_tool_end 分支**（现有 `card` 处理在 L529–576）：在 LangChain 流事件循环里增加对 `recommendItinerary` 的判断分支：
   - 解析工具返回的 JSON 字符串为 `Itinerary` 对象；
   - 复用现有的"单轮去重"机制（`emittedPublicTools` / 类似集合）保证一轮最多 emit 一次；
   - 调 `emitEvent('itinerary', { itinerary })`。
3. **过滤公开工具列表**：现有逻辑里 `finalizeTripCard` 被过滤，不下发 `tool_start` / `tool_end`；同样把 `recommendItinerary` 加进过滤集合，保持"内部工具"语义。
4. **互斥保证**：在同一轮内若已 emit `card`，则跳过 `itinerary`（反之亦然），避免单轮同时渲染两张卡。具体实现：复用现有 `cardEmitted` 旗标，新增 `itineraryEmitted`，互斥判断。

### 2.5 Prompt 触发条件

修改 [apps/server/src/agent/prompts.ts](../../apps/server/src/agent/prompts.ts) 规则 3 / 规则 4：

- 在规则 3 后追加**新触发分支**："当用户明确要求规划行程（如『X 三天两晚怎么规划』『五一去 X 玩 N 天』『带老人去 X 两天怎么安排』），且给出了**单一目的地**时，调用 `recommendItinerary` 工具一次，参数按照 schema 填充逐日活动；调完后**直接以空字符串结束本轮回复**，不要再输出 markdown，前端会渲染行程卡。"
- 明确**与 TripCard 工具链互斥**：当本轮意图是规划行程时，**不要**再调 getWeather / getAttractions / finalizeTripCard；当本轮意图是"想看 X 城市的天气和景点"时，走原 TripCard 三工具链，不要调 recommendItinerary。
- 在规则 4 的"基于已有卡片的追问"中补充：消息历史里若包含 `[已为「xxx」生成行程规划...]` 摘要，本轮针对该地行程的追问（"第二天能不能换成西溪湿地"、"老人能不能省略灵隐"）一律走自然语言回答，不重复调工具。
- 同步更新 chat.ts 里给消息历史注入的"已生成卡片"摘要文案（现有逻辑里 TripCard 注入的是 `[已为「xxx」生成完整行程卡...]`，行程卡用 `[已为「xxx」生成行程规划...]` 区分）。

### 2.6 字段约束硬底线（防编造）

工具 schema 注释里强调：
- 不要编造门票价格、酒店名称、班次时刻等动态数据；
- 不要为不熟悉的小众目的地虚构景点；
- 时间字段优先用"上午 / 下午 / 傍晚 / 晚上"宽口径，不用"08:30"这种精确点；
- 备选活动用 `time: "或"` 前缀，与设计稿一致。

## 三、Frontend 技术方案

### 3.1 类型与 SSE 事件接入

`@travel/shared`（或现有 alias）已自动导出 `Itinerary` / `ItineraryDay` / `ItineraryItem`，前端直接 import。无需新建本地类型。

修改 [apps/web/src/chat/useTravelAgent.ts](../../apps/web/src/chat/useTravelAgent.ts) 的 SSE 事件循环（现有 token / tool_start / tool_end / card / final 五分支，L213–273）：

- 新增 `case 'itinerary'` 分支，把 payload 里的 `itinerary` 字段 patch 到当前 assistant 消息：`message.itinerary = payload.itinerary`。
- 在 assistant 消息类型上扩展可选字段 `itinerary?: Itinerary`（与 `card?: TripCard` 平级）。
- 现有 `hasStructured` 判断扩展为 `Boolean(message.card || message.itinerary)`，保证有任一卡片时切到结构化渲染分支，否则才 fallback 到 MarkdownTyping。

### 3.2 ItineraryCard 组件

新建 [apps/web/src/chat/cards/ItineraryCard.tsx](../../apps/web/src/chat/cards/ItineraryCard.tsx)，按照 [Itinerary-planning.html](Itinerary-planning.html) 内的 `ItineraryCard` 函数照搬：

- 顶部 header：左侧 `<I.map />` icon + 宋体 `title`（fallback "行程规划"），右侧 mono `{days.length} DAYS · ITINERARY`，底部 1px solid 分隔。
- 主区：`grid-template-columns: 78px 1fr` 左右分栏；左侧 rail 显示 `DAY N` + `subtitle`，右侧时间轴。
- 时间轴：右侧绝对定位 1.5px 竖线 `var(--card-line)`；每个 item 用 `grid-template-columns: 16px 1fr`，圆形空心节点（`border: 2px solid var(--accent)`）+ 主体；主体内顶行排列 `time`（mono）/ `title`（粗体）/ `tag`（accent 背景白字 pill）；次行 `desc` 灰字。
- 底部 `footnote` 区：虚线分隔（`border-top: 1px dashed var(--card-line)`），加粗"提示："前缀 + 内容。整段 `footnote` 缺失时不渲染。
- 颜色变量沿用 TripCard 已有的 `--card` / `--card-line` / `--card-line-soft` / `--card-ink` / `--card-ink-soft` / `--card-ink-mute` / `--accent` / `--accent-hl`。如这些 CSS variable 在前端样式入口尚未声明，到样式入口（按 TripCardView 的引用方式查找现有 CSS 变量定义文件）补齐。
- icon `I.map` 单独抽到本组件文件顶部或与 TripCard 共享的 icon 模块。

### 3.3 渲染分发

修改 [apps/web/src/chat/cards/TripCardView.tsx](../../apps/web/src/chat/cards/TripCardView.tsx) 邻近的"消息渲染分发"位置（实际入口在 ChatPage.tsx L260 附近）：

- 当 `message.itinerary` 存在时，渲染 `<ItineraryCard {...message.itinerary} />`，与 TripCardView 平级互斥。
- 优先级：`card` 与 `itinerary` 同时存在时（按设计不会发生，但防御性处理），按出现先后渲染只保留一张；推荐以"哪个先到先渲染"为准，或简单按 `card` 优先级。

### 3.4 fallback 与空态

- 工具未返回 `footnote` 时整块不渲染；
- `days[].items` 为空数组时不应该出现（schema 限制 ≥ 1）；防御性渲染：若 `items.length === 0` 整个 day 不渲染；
- `tag` 缺失时只展示 `time` + `title`，不留空白 pill。

### 3.5 验证（人测用例）

启动 dev server（仓库约定脚本），逐条手测：

| 用例 | 期望事件流 | 期望渲染 |
|---|---|---|
| "杭州三天两晚怎么规划？" | 仅 `itinerary` 事件，无 `card` | ItineraryCard 显示 3 天日程，标题含"杭州"和"三天两晚" |
| "五一去成都玩 4 天" | 仅 `itinerary` | ItineraryCard 4 天，标题含"5/1"与"成都" |
| "带老人去苏州两天怎么安排比较轻松" | 仅 `itinerary` | ItineraryCard 2 天，节奏明显放慢，footnote 含老年友好提示 |
| **回归**："我想去厦门" | `tool_start`/`tool_end`/`card` | TripCard 渲染（不应触发 itinerary） |
| **回归**："海南美食攻略有吗？" | 纯 message token 流 | markdown 文本，无任何卡片 |
| 已生成行程后追问"第二天能不能换成西溪湿地" | 纯 message token | 自然语言回答，不再 emit `itinerary` |

观察重点：
1. SSE 事件序列正确，单轮内 `card` 与 `itinerary` 不同时出现；
2. 行程卡视觉与 [Itinerary-planning.html](Itinerary-planning.html) 设计稿一致（DAY rail、时间轴节点、accent 色、宋体标题、虚线 footnote）；
3. LLM 不能编造门票价格、酒店班次等非范围内细节；
4. 已有 TripCard 用例不被破坏。
