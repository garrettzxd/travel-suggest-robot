# 放宽「漫游助手」角色边界 — 实施计划

## Context

当前 bot 在用户问截图里这四类问题时一律走拒绝模板：
1. "海南美食攻略有吗？"
2. "5/1 雨天室内有什么推荐？"
3. "带老人怎么安排行程？"
4. "三亚和海口哪个更好玩？"

根因都在后端 system prompt：[apps/server/src/agent/prompts.ts](apps/server/src/agent/prompts.ts) L11-14 把"角色边界"写死成"只回答景点/天气/时节"，L41 又强制"多城市必须二选一"。架构层（LangGraph 工具集、SSE 路由）没有硬约束 —— 改动**只动 prompt 文案**就能放宽。

目标：把"只能查景点+天气+时节"扩展为"全方位旅行规划顾问"，覆盖美食、行程编排、人群适配（老人/亲子）、雨天/极端天气替代方案、城市对比、无地名的开放性目的地建议；同时**保留三工具链**用于"用户明确想去某地"的场景，避免 TripCard 渲染逻辑变动。

## 核心改动

**唯一需要修改的文件**：[apps/server/src/agent/prompts.ts](apps/server/src/agent/prompts.ts)

### 1. 重写「角色边界」（覆盖原 L7、L11）

把首段定位从"专注的旅游规划助手，只回答景点/天气/时节"改为"旅行规划顾问，覆盖目的地选择、景点、天气、美食、行程编排、人群适配、对比建议等"。

明确**仍属于范围内**的话题（举例）：
- 景点、玩法、目的地推荐、城市/景点对比
- 天气、时节、雨天/恶劣天气下的室内或替代方案
- 美食与饮食特色（不需要查实时数据时直接基于常识回答）
- 行程节奏、天数安排、带老人/带娃/无障碍等人群适配
- 交通与住宿大方向建议（不做具体订位、不编造价格/班次）

明确**仍要拒绝**的：编程、数学、写作、通用知识闲聊、与旅行完全无关的请求。

### 2. 重写「拒绝模板」触发条件（覆盖原 L12-14）

只在用户问的是**完全与旅行无关**的内容时才使用拒绝模板。把当前 L14 的措辞从"我只能帮你规划旅行（景点推荐、天气查询、出行时节建议）"改为更开放的版本，例如：

> 「抱歉，这超出了我的旅行规划能力范围。如果你想出去玩，告诉我你感兴趣的城市、出发时间或想要的体验，我可以帮你看天气、推荐景点、安排行程。」

打招呼模板（L16-24）保持不变。

### 3. 重写「工具调用规则」触发条件（覆盖原 L26、L40）

将"一旦提到任何具体地名就必须并行调用 getWeather + getAttractions"改为按**用户意图**判定：

- **触发三工具链的场景**：用户明确表达"想去 X / 准备去 X / 帮我看 X 怎么样 / X 这周适不适合去 / X 天气如何"等"想要 X 城市的实时数据 + 行程卡"的诉求，且 X 是单一明确地名。此时按原来的三步流程（并行 getWeather + getAttractions → finalizeTripCard → 空文本结尾）执行。
- **不触发工具链的场景**（**直接用自然语言回答，本轮不调任何工具，不渲染 TripCard**）：
  - 美食、行程编排、人群适配、室内/雨天替代方案等不需要实时景点/天气检索的咨询；
  - 多城市对比题（如"三亚 vs 海口"）—— 用常识做对比，并在结尾引导用户选定一个再帮看天气和景点；
  - 没有具体地名的开放性问题（如"5 月国内带老人去哪好"）—— 给出 2–3 个候选目的地，每个一句话理由，引导用户选定一个再深入；
  - 上下文里**已经有过 TripCard** 的追问（消息历史里包含 `[已为「xxx」生成完整行程卡...]` 摘要），如"5/1 雨天室内有什么推荐？""带老人怎么安排"——默认基于该地，用自然语言回答，**不重复调工具**。

### 4. 改写「多城市规则」（覆盖原 L41）

从"用户提到多个城市时请回复请求他二选一"改为：

- 若用户是在做**对比**（"X 和 Y 哪个更好玩"、"X vs Y"），允许直接用自然语言比较两地特色，结尾引导选定一个再帮深查；
- 单轮工具链仍然只为一个目的地工作；如果用户对比之后明确选了一个，再走三工具链。

### 5. 保留不变

- L18-24 打招呼模板（强一致 UI 文案）；
- L29-33 finalizeTripCard 的字段约束、verdictBadge / chips / 文案对照表；
- L36-39 反编造、不重复 finalize、不罗列 JSON 等"工具链一旦触发要遵守"的规则；
- L42 风格段。

## 不需要改动的文件

- [apps/server/src/agent/graph.ts](apps/server/src/agent/graph.ts)：工具集仍是 getWeather / getAttractions / finalizeTripCard，不引入新工具。
- [apps/server/src/routes/chat.ts](apps/server/src/routes/chat.ts)：SSE 翻译、dedup、TripCard emit 逻辑都不动 —— 因为新规则下"对比/美食/行程"等问题根本不调工具，自然不会触发 TripCard。
- 前端：纯文本回复走原有 message token 流，已经能正常渲染。

## 验证

启动 dev server（`pnpm dev` 或仓库约定脚本），人工跑一遍以下用例：

| 用例 | 期望 |
|---|---|
| "海南美食攻略有吗？" | 自然语言美食概览（文昌鸡、清补凉等），无 TripCard |
| "5/1 雨天室内有什么推荐？"（前面已问过北京） | 基于北京上下文给室内/雨天替代方案，不重复 getWeather/getAttractions |
| "带老人怎么安排行程？"（前面已有目的地） | 自然语言行程节奏建议，结合上下文 |
| "三亚和海口哪个更好玩？" | 文本对比 + 引导用户选定一个，无 TripCard |
| "5 月国内带老人去哪好？" | 给 2–3 个候选目的地，每个一句话理由 |
| "我想去厦门" | **照常**走三工具链 + TripCard（回归用例，不能被破坏） |
| "帮我写代码" | **照常**走拒绝模板（回归用例） |
| "你好" | **照常**走打招呼模板，逐字一致（回归用例） |

观察重点：
1. 不该调工具的回答里 SSE 流应**没有** `tool_start` / `tool_end` / `card` 事件；
2. 对比题与开放题里 LLM 不能编造温度、降水、评分等数据；
3. "我想去厦门"等触发用例里 TripCard 仍能正常渲染、`verdictBadge` 与 `recommendation.tag` 对齐。
