# 聊天流式更新强制滚动到底部

## 问题表现

- 场景位于 Web 聊天页 `apps/web/src/chat/ChatPage.tsx`，TripCard 在流式接收天气、景点、地点、总结等结构化数据时会多次更新同一条 assistant 消息。
- 用户把页面停在天气卡片附近查看内容时，天气卡片位于屏幕垂直方向中部，滚动条并没有置底。
- 每次 chat 接口收到新的 SSE 数据并 patch `messages` 后，页面都会自动滚动到底部，导致用户正在查看的天气卡片被移出当前视窗。
- 影响集中在体验层面：数据加载过程中无法稳定阅读中间卡片，尤其是渐进式卡片事件增多后，抢滚动位置的现象更明显。

## 问题原因

1. 原滚动逻辑绑定在 `messages` 变化上，只要消息数组发生任何更新，就无条件执行 `el.scrollTop = el.scrollHeight`。
2. 渐进式 TripCard 会在 `tool_end`、`card_destination`、`card_weather`、`card_attractions_summary` 等事件到达时多次 patch 当前 assistant 消息；这些更新都会触发 `messages` 变化。
3. 原逻辑没有区分“用户仍贴近底部，期望自动跟随”与“用户已上滑查看中间内容，期望保持当前位置”两种状态，因此后续数据到达时会强行把滚动条拉到底部。
4. 修复点位于 [`apps/web/src/chat/ChatPage.tsx:117`](../apps/web/src/chat/ChatPage.tsx:117) 到 [`apps/web/src/chat/ChatPage.tsx:148`](../apps/web/src/chat/ChatPage.tsx:148)：新增 `shouldStickToBottomRef`，用滚动距离判断是否继续自动跟随。

## 解决方法

- 在 `ChatPage` 中新增 `shouldStickToBottomRef`，记录用户是否仍处于底部附近。
- `useEffect([messages])` 不再无条件滚动到底部；只有 `shouldStickToBottomRef.current === true` 时才执行自动滚动。
- 给滚动容器增加 `onScroll`，通过 `scrollHeight - scrollTop - clientHeight` 计算距离底部的像素值，小于 `96px` 时认为用户仍在底部附近。
- 用户提交输入、点击 Welcome 建议或点击 chips 发起新请求时，将 `shouldStickToBottomRef.current` 重置为 `true`，保证新一轮对话仍会主动跟随最新输出。

## 验证方式

- 执行类型检查：

```bash
pnpm --filter @travel/web exec tsc --noEmit
```

- 手动验证：
  - 发起一个会触发 TripCard 的请求，例如“我想去杭州”。
  - 在天气卡片出现后上滑或停留，让天气卡片位于视窗中部且滚动条不在底部。
  - 等待后续 `card_weather` / `card_attractions_summary` 等数据到达。
  - 预期：卡片内容继续更新，但页面不再自动滚动到底部。
  - 再次发送新消息或点击 chips。
  - 预期：新一轮请求开始后页面恢复自动跟随到底部。
