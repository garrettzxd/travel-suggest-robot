# question-record

为 travel-suggest-robot 项目生成问题记录文档。目标读者是未来的自己或同事，**能冷启动读懂**这次问题是什么、为什么发生、怎么修的。

## 什么时候触发

用户在一次排障 / 修 bug / 联调解决之后，表达出"记录下来"、"归档"、"写一份问题记录"、"生成 question doc"等意图。触发时**不要再复述结论**，直接产出文档。

## 产出约定

- **落地路径**：`questions/<kebab-case-标题>.md`（相对仓库根目录）。
- **文件名**：使用中文名称，文件名称与正文标题保持一致
- **写作语言**：正文中文。代码块、路径、日志英文原样保留。
- **不要**写 emoji（排查清单里的 ❌/✅ 除外）。
- **不要**在文档里写"本次对话"、"刚才的修复"之类的语境黑话——冷读者读不懂。
- **写作视角**：事后复盘视角，已定稿，不是对话流。

## 文档结构（模板）

```markdown
# <问题的一句话标题>

## 问题表现

- 场景前置（环境、栈、请求路径等）。
- 具体观察到的异常（错误信息原文、UI 现象、日志片段），贴原始内容而不是转述。
- 区分"功能层面"和"体验/调试层面"的影响。

## 问题原因

- 从证据链出发，按 1/2/3 点列出排查结论。
- 关键代码用仓库相对路径 + 行号引用：`[apps/server/src/routes/chat.ts:305](../apps/server/src/routes/chat.ts:305)`。
- 解释清楚「为什么会出现这个现象」，不是只说"改了 X"。
- 如果牵扯到第三方行为（LangChain 转换器、Kimi API、Chrome DevTools 等），说清楚触发条件。

## 解决方法

- 最终采用的方案（贴关键 diff 片段或改动位置）。
- 为什么这样修（对比备选方案的取舍，如果有）。
- 必要的部署 / 重启 / 清缓存步骤。

## 验证方式

- 可复现的验证步骤（curl / 点击路径 / 观察指标）。
- 预期结果的明确判据。

## 附带排除过的可能性（可选）

以 ✅ / ❌ 列表记录走过的死胡同，避免后续重复踩。

- ❌ <某个假设> —— 证伪证据。
```

**三段必写**：问题表现 / 问题原因 / 解决方法。其它小节（验证方式、排除过的可能性、引用链接）按材料丰俭决定。

## 生成流程

1. **确定标题和文件名**：从对话上下文提取核心关键词，转成 kebab-case 英文文件名。先 `Read` 或 `Glob` 看一眼 `questions/` 有没有重名，有就换名或追加日期后缀。
2. **整理材料**：从本次对话回溯排障链路：用户最初的错误现象 → 中途试过的假设 → 定位到的根因 → 最终改动。把代码改动里的文件路径和行号记下来备用。
3. **写入文档**：用 `Write` 创建文件，按模板填充。对每段保持"证据先行"——贴日志、贴原始错误、贴 response 片段，而不是只写结论。
4. **交付**：最后用 markdown 链接把文件路径告诉用户（`[questions/xxx.md](questions/xxx.md)`），一句话概括覆盖范围。不需要再复述文档内容。

## 风格要点

- 代码引用一律 markdown 链接 + 行号：`[chat.ts:337](apps/server/src/routes/chat.ts:337)`。
- 路径用仓库根目录的相对路径，不要绝对路径。
- 错误信息、响应片段原样引用，不要改动标点或截断关键字段。
- 每段控制在 5-8 条列表项以内，超了就拆小节。
- 如果问题原因涉及"第三方库内部行为"，必要时指向 `node_modules/.pnpm/...` 的具体文件路径，方便二次求证。
